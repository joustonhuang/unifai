# Codex Contribution Policy

This repository allows AI-assisted development using tools such as GPT Codex.
However, AI must operate within a controlled engineering workflow.

Codex is treated as a contributing engineer, not an autonomous maintainer.

All changes must follow the governance rules below.

---

# 1. Engineering Philosophy

UnifAI is developed using a **slow, controlled, human-reviewed workflow**. This allows a better, 
secured path of development

Our development cycle is intentionally conservative:

1. Make a small change.
2. Human reviews the diff.
3. Human tests the change.
4. Only then proceed to the next step.

Large refactors without explicit approval are not allowed.

AI tools must respect this workflow.

---

# 2. Codex Role

Codex is expected to function as a **high-skill engineering assistant**.

Typical responsibilities include:

- generating small patches
- refactoring localized code
- updating configuration references
- writing test scripts
- explaining existing code
- proposing improvements

Codex **does not have authority to redesign the system architecture**.

Architectural changes must be proposed, not implemented automatically.

---

# 3. Allowed Change Scope

Codex may directly implement:

- mechanical refactors
- filename normalization
- configuration updates
- documentation clarifications
- small bug fixes
- test utilities
- installer robustness improvements

Changes should normally affect **a small number of files**.

If a change touches many files, Codex must first explain why and ask for approval.

---

# 4. Restricted Changes

Codex must NOT automatically modify:

- system architecture
- governance rules
- agent role definitions
- world charter semantics
- protocol design
- security boundaries
- resource governance policies

Examples in this repository include:

- `world_charter.yaml`
- agent role definitions
- governance logic
- runtime architecture layers

These areas require human architectural review.

Codex may **suggest changes**, but must not apply them directly.

---

# 5. Diff Discipline

Every Codex task should follow this pattern:

1. Analyze repository state
2. List files that will change
3. Explain why each change is required
4. Apply the smallest possible patch
5. Save a summary with diff in this pass into ./tmp/ preferrably using same filename with timestamp.

Codex should aim to produce the **smallest meaningful diff**.

Unnecessary rewrites are discouraged.

---

# 6. Documentation Safety

Codex must not rewrite major documents unless explicitly requested.

Protected documents include:

- `README.md`
- `AGENTS.md`
- governance documents
- architecture explanations

Codex may correct typos or clarify wording if explicitly asked.

Large documentation rewrites require human approval.
